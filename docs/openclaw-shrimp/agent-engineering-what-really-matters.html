Agent 工程里，真正拖后腿的不是模型

来源：本文精炼自 tw93 的《你不知道的 Agent：原理、架构与工程实践》。原文信息密度极高，值得完整阅读。这里只摘录我认为最值得记住的几个工程判断。

tl;dr：更贵的模型带来的提升很多时候没有想象中那么大。真正决定系统能不能稳定运行的，是 Harness、上下文工程和工具设计这几块外围工程条件。

Agent Loop 本身并不复杂

Agent Loop 抽象出来不到 20 行代码：感知→决策→行动→反馈，循环往复，直到模型返回纯文本。

看过不少 Agent 实现，结构大同小异。主循环本身相当稳定，从最小实现一路扩展到子 Agent、上下文压缩和 Skills 加载，核心逻辑几乎没有变化。新增能力通常叠加在循环外部，而不是改动循环内部。

模型负责推理，外部系统负责状态和边界。这个分工一旦确定，核心循环很少需要频繁调整。

Harness 比模型更关键

Harness 是指围绕 Agent 构建的测试、验证与约束基础设施，至少包括四个部分：验收基线、执行边界、反馈信号和回退手段。

OpenAI 内部有一个数据：3 个工程师 5 个月写了百万行代码，将近 1500 个 PR，是传统开发速度的 10 倍。这个速度靠的不是模型有多强，而是几个工程决策做对了：

  • 约束编码化而非文档化：写在文档里的规范容易被忽略，编码进 Linter、类型系统或 CI 规则里的约束才具备可执行性
  • 端到端自主：查日志、查指标、查追踪，都由 Agent 主动完成，不需要人告知错误
  • 最小化合并阻力：测试偶发失败用重跑处理，而不是阻塞进度

一个判断值得记住：Harness 要做的事情，是把任务推进"目标明确、结果可以自动验证"的区域，让对错有机器可以执行的判断标准，而不是靠人盯。

上下文分层是稳定性的根因

Transformer 的注意力复杂度是 O(n²)，上下文越长，关键信号越容易被噪声稀释。实践里最常见的失效模式是：无关内容一旦占到上下文的大头，Agent 的决策质量就会明显下滑。这类现象通常被叫作 Context Rot，很多看起来像模型能力不足的问题，往往可以追溯到上下文组织不当。

解决方式是按信息的使用频率和稳定性分层管理：

  • 常驻层：身份定义、项目约定、绝对禁止项，保持短、硬、可执行
  • 按需加载：Skills 和领域知识，描述符常驻，完整内容触发时再注入
  • 运行时注入：当前时间、渠道 ID 等动态信息，每轮按需拼入
  • 记忆层：跨会话经验写入 MEMORY.md，不直接进系统提示，需要时才读取
  • 系统层：Hooks 或代码规则处理确定性逻辑，完全不进上下文

别把确定性逻辑放进上下文。凡是可以通过 Hooks、代码规则或工具约束表达的内容，都应交给外部系统处理，而不是让模型反复读取。

Skills 按需加载的精髓

Skills 是上下文工程里非常有效的一种模式，核心思路是：系统提示只保留索引，完整知识按需加载。

Skill 描述有两个写法陷阱：

第一个是字数。低效的描述约 45 tokens，高效的描述约 9 tokens。路由准确率差距不大，但每个启用的 Skill 描述符都常驻上下文，Skill 一多，长描述的累积成本很可观。

第二个是精度。描述太短等于任何相关工作都能触发，路由会乱。真正有效的描述符是路由条件，不是功能介绍——"何时该用我"比"我能做什么"重要得多。

一个反直觉的地方：稳定的大系统提示，比频繁变动的小提示实际成本更低。写入成本只付一次，后续每次调用读取的折扣可以达到 90%。

工具设计决定 Agent 能做什么

上下文决定模型能看到什么，工具决定模型能做什么。工具定义的质量比数量更关键。仅 5 个 MCP 服务器就可能带来约 55,000 tokens 的工具定义开销，相当于在 200K 上下文里还没开始对话就用掉了近三成。

工具问题多数不在数量不够，而在选不对、描述看不懂、返回一堆没用的、出了错 Agent 也不知道怎么改。

工具设计大致经历了三个阶段：

  • 第一代，API 封装：每个 API Endpoint 对应一个工具，粒度过细
  • 第二代，ACI（Agent-Computer Interface）：工具对应 Agent 要完成的目标，而不是底层 API 操作
  • 第三代，Advanced Tool Use：动态工具发现、代码编排、示例驱动

ACI 原则里最关键的一条：结构化错误要给出修正建议，而不是只返回 "Error: update failed"。Agent 报错后能否快速修回来，和工具设计的关系更大。

记忆系统是基础设施，不是事后补上的

Agent 不具备原生的时间连续性。会话结束后，上下文随之清空，下一次启动时也不会自动保留此前状态。要让系统具备跨会话的一致性，记忆层得单独设计。

四种记忆各司其职：

  • 上下文窗口：工作记忆，token 有限，得主动管理
  • Skills：程序性记忆，按需加载不默认常驻
  • JSONL 会话历史：情景记忆，磁盘持久化，支持跨会话检索
  • MEMORY.md：语义记忆，Agent 主动写入的稳定事实，每次启动时注入

记忆整合最关键的不是摘要写得多漂亮，而是流程本身必须可回退。系统只移动指针，不删除原始消息，即使整合失败，也还能回到原始存档继续工作。

原文结语里的判断值得直接引用：工程上把这些做好，不靠砸钱上最新模型，Agent 的稳定性和实际产出就能上一个台阶。模型是加速器，但基础设施才是地基——这个判断在代码生成这类高可验证任务上最成立，也最容易被忽视。
