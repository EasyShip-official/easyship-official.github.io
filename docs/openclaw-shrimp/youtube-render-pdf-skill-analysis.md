## 一、Skill 是什么

把一个 YouTube 技术讲座视频，自动转换成一份完整的、可编译的中文 LaTeX 讲义和 PDF。

**输入：** YouTube URL  
**输出：** 封面图 + `.tex` 文件 + 所有图资源 + 编译好的 PDF

---

## 二、为什么这个 Skill 成功了

### 2.1 约束即自由

大多数 Skill 的错误是给模型太多自由度，说"生成一份好的讲义"——模型面对无边界任务，不知道什么算好。

这个 Skill 的做法：**把"好讲义"拆解成原子级具体规则，每条规则对应一个可验证的产出要求。**

约束不是限制，是框架。

### 2.2 三盒系统的教学设计

最核心的创新——三种盒子，各有明确认知功能：

```
\importantbox{}  — 核心概念
  用于：形式定义、中心论点、关键机制总结、定理性陈述、核心算法步骤

\knowledgebox{}   — 背景知识
  用于：先修提醒、历史脉络、工程背景、设计权衡、术语对比

\warningbox{}    — 常见误区
  用于：符号过载、隐藏假设、误导性启发式、易犯错误
```

**为什么有效：**

- 盒子有使用触发条件，不是随意添加
- 每种盒子对应一种认知功能，不是装饰
- 数量不设上限——材料有多少高价值信号，就用多少盒子
- 盒子和正文严格分离——图片不能放盒子里

### 2.3 帧选取协议：精准召回的方法论

大多数 AI 视频总结的错误：基于字幕猜测时间戳，选一个帧就算完成。

这个 Skill 的帧选取是**可执行的精确流程**：

```
第一步：基于字幕时间戳定位，而不是猜测
  字幕 span → 对应时间区间 → 在区间内搜索 → 比较多个候选帧 → 选最优

第二步：先看内容，再命名
  原始帧：frame_001.jpg（时间戳命名）
  ❌ 没看之前就叫"架构图"
  ✅ 视觉确认是"三层微服务拓扑图" → 重命名为 microservices-topology.jpg

第三步：渐进揭示找最终状态
  PPT 动画 / 白板累积 / 仪表盘
  → 不要停在第一个近似帧
  → 继续搜索，找到内容最完整、最可读的那一帧
```

### 2.4 交付清单：把"完成"变成可验证的

```
必须交付：
✅ 最终 .tex 文件
✅ 下载的封面图
✅ 所有引用的图资源文件
✅ 编译好的 PDF（不是"写完 tex 就行"）
```

**关键区别：**
- 生成内容：说"完成了"
- 交付物：说"东西在这里，可以用了"

## 三、与传统 Prompt Engineering 的本质区别

| 维度 | 传统 Prompt | This Skill |
|------|------------|-----------|
| 成功标准 | 模糊（"生成好内容"） | 具体（编译成功 + 所有文件交付） |
| 约束方式 | 负面约束（"不要做什么"） | 正面约束（明确触发条件） |
| 盒子使用 | 无 | 三种盒子各有用途 |
| 帧选取 | 猜测 + 近似 | 字幕时间戳定位 + 验证清单 |
| 输出定义 | 内容 | 交付物清单 |

## 四、对 Skill 设计的核心启示

### 约束式 vs 规定式

```
规定式（错误）：
"第一步做 X → 第二步做 Y → 第三步做 Z"
→ agent 是执行器

约束式（正确）：
"当 X 情况出现，用 Y 方式处理"
"当 Z 情况出现，用 W 方式处理"
"目标达成时，交付 A"
→ agent 是决策者
```

这个 Skill 是约束式 harness 的完整示范。

### 三盒系统可直接迁移

你的 `/new-project`、`/review`、`/handoff` Skill 可以用同样的盒子逻辑：

```
/new-project 中的 \importantbox：
  - 领域边界定义
  - 核心 bounded context
  - 成功的可验证标准

/new-project 中的 \warningbox：
  - 已知的无效方向
  - 常见的需求范围蔓延模式
  - 模型容易偏离的触发点

/new-project 中的 \knowledgebox：
  - 类似项目的参考案例
  - 技术选型的权衡分析
```

### 交付清单 = 验收协议

每个 Skill 的结尾应该是明确的交付清单，而不是模糊的"任务完成"。

## 五、对 manim 动画项目的启发

| 这个 Skill 的做法 | 对 manim 的迁移 |
|---|---|
| 三盒系统（重要/背景/误区） | 三级视觉强调：核心动画 / 辅助图示 / 警示动画 |
| 帧选取用字幕时间戳定位 | 动画节点用"讲解节奏"定位 |
| 必须编译成功 PDF | 必须 render 成功出视频 |
| 交付物是 PDF，不是 tex 源文件 | 交付物是视频，不是 .py 源文件 |
| 时来源记录（00:12:31--00:12:46） | 动画时间轴注释：每个场景对应哪个讲解节点 |

**最核心的迁移原则：**

> 不要先写文案再配动画。要先确定"什么动画能最直观地展示这个概念"，再让文案去配合动画的节奏。

动画是表达的主体，文案是解释语言，不是相反。
